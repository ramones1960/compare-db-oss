💡 Cache-Aside（キャッシュ退避）パターン: アプリがキャッシュを管理する。 Hit 時はキャッシュから返す（超高速）。Miss 時は DB から取得してキャッシュに書き込む。 読み取り負荷が高いシステムで DB の負荷を大幅に削減できる。 TTL でキャッシュの鮮度を管理。書き込み時は明示的にキャッシュを invalidate する必要がある。

EC サイトの注文フローを「最適な DB」で担当させます。各ステップの処理時間を確認できます。

ClickHouse 注文イベントを分析 DB に記録 —
💡 Polyglot Persistence パターン: 用途ごとに最適な DB を選ぶ。 検索 → OpenSearch、キャッシュ/セッション → Redis、トランザクション → PostgreSQL、分析 → ClickHouse。 単一 DB では「何でもできるが何も最速でない」状況を回避できる。 ただし運用コストが増えるため、最初は 1〜2 DB から始めることを推奨。

💡 CQRS（コマンドクエリ責務分離）パターン: 書き込みと読み取りのモデルを完全に分離する。 Write 側は正規化・ACID・整合性優先。Read 側は非正規化・集計最適化・速度優先。 OLTP + OLAP の役割分担で、両方のワークロードを最適化できる。 CDC（Change Data Capture）でリアルタイムに Read 側へ同期する構成が一般的。
