# Cache-Aside パターン

| 項目 | 内容 |
|---|---|
| 構成DB | PostgreSQL（メインDB）+ Redis（キャッシュ） |
| ポート | PostgreSQL: 5432 / Redis: 6379 |
| 難易度 | ★☆☆ 入門 |
| キーワード | キャッシュ、低レイテンシ、読み取り最適化 |

## パターンの概要

**Cache-Aside**（別名: Lazy Loading）は、アプリケーション自身がキャッシュの管理責任を持つ、最も広く使われるキャッシュパターンです。

「キャッシュを脇に置く（aside）」という名前の通り、メインDBとキャッシュは独立して存在し、アプリが仲介役として両方を操作します。

## アーキテクチャ図

```
         ┌─────────────────────────────────────────────────────────┐
         │                    Application                          │
         │                                                         │
         │  ┌──────────────────────────────────────────────────┐   │
         │  │              読み取りロジック                     │   │
         │  │                                                   │   │
         │  │  1. Redis にキャッシュを確認                     │   │
         │  │       HIT ──→ キャッシュから返す ✓              │   │
         │  │       MISS ─→ 2. DB から取得                    │   │
         │  │               3. Redis にキャッシュ格納          │   │
         │  │               4. 値を返す                        │   │
         │  │                                                   │   │
         │  │  書き込みロジック                                 │   │
         │  │       1. DB に書き込む                           │   │
         │  │       2. Redis のキャッシュを削除（Invalidation）│   │
         │  └──────────────────────────────────────────────────┘   │
         └──────────────────────┬──────────────────┬───────────────┘
                                │                  │
                   ┌────────────▼───┐    ┌─────────▼────────┐
                   │  PostgreSQL    │    │     Redis          │
                   │  (メインDB)    │    │   (キャッシュ)     │
                   │  ポート: 5432  │    │   ポート: 6379     │
                   │                │    │   TTL: 60秒        │
                   └────────────────┘    └──────────────────┘
```

## 読み取りフロー（Cache-Aside）

```
App ──[GET product:1]──→ Redis
         ╔═══════╗
    HIT  ║       ║ MISS
         ╚═══════╝
          │           │
          ▼           ▼
       返す      PostgreSQL
                    │
                    ▼
              Redis に SET
              (TTL 60秒)
                    │
                    ▼
                 返す
```

## デモの内容（demo.py）

1. PostgreSQL に商品テーブル作成 + データ挿入
2. **キャッシュ MISS** → DB から取得 → Redis にキャッシュ格納
3. **キャッシュ HIT** → Redis から高速取得
4. 在庫更新 → **キャッシュ無効化**（DELETE）
5. 再度 MISS → 最新値を DB から取得

## いつ使うか

- **読み取り頻度が高く、書き込み頻度が低い**データ（商品マスタ、ユーザープロフィール等）
- レイテンシを数ms以下に抑えたい API レスポンス
- 同じデータを短期間に何度も参照するワークロード
- まずはキャッシュを試したいという場面（シンプルで導入しやすい）

## いつ使わないか

- 書き込みが多く、キャッシュが頻繁に無効化される場合（効果が薄い）
- データの鮮度が厳密に要求される場合（キャッシュ更新前の古い値が問題になる）
- キャッシュの状態管理をアプリ側に持ちたくない場合（→ Write-Through / Read-Through を検討）

## 注意点

### キャッシュ無効化（Cache Invalidation）

「キャッシュの無効化はコンピュータサイエンスで最も難しい問題の一つ」と言われます。

| 戦略 | 説明 | メリット | デメリット |
|---|---|---|---|
| 書き込み時に削除 | 本デモの方式。更新後に `DEL` | シンプル | 次回アクセスが必ず MISS |
| 書き込み時に更新 | 更新後に `SET` でキャッシュ更新 | 即座に整合 | 並行更新で古い値が残る可能性 |
| TTL のみ | キャッシュは TTL 切れで自然消滅 | 実装簡単 | TTL 中は古いデータが返る |

### Thundering Herd（サンダリング・ハード）問題

人気コンテンツのキャッシュが切れた瞬間、大量のリクエストが同時に DB へ押し寄せる現象。

```
キャッシュ期限切れ
    │
    ├── リクエスト1 → DB へ（取得中...）
    ├── リクエスト2 → DB へ（取得中...）
    ├── リクエスト3 → DB へ（取得中...）
    └── ...（DB に集中負荷）
```

**対策**:
- **Mutex Lock**: 最初の1リクエストだけ DB に問い合わせ、他は待機
- **Probabilistic Early Expiration**: TTL 切れ前にランダムに事前更新
- **Stale-While-Revalidate**: 古い値を返しつつバックグラウンドで更新

### キャッシュ汚染（Cache Pollution）

存在しないキーへの問い合わせが多い場合（Cache Penetration）、毎回 DB が叩かれます。

**対策**: 存在しない場合も `null` をキャッシュする（Null Object キャッシュ）か、Bloom Filter で事前フィルタ。
